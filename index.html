What is a Tree?

A tree is a connected structure where each element (called a node) holds a value and can point to child nodes. There's exactly one starting node — the root — and no cycles allowed.

Real-world analogy

Think of a company org chart. The CEO is at the top (root). The CEO has direct reports (children). Each of those has their own reports. Nobody reports to two different managers (no cycles). That's a tree.

Types of Trees

Binary Tree

Each node has at most 2 children — called left and right. This constraint unlocks powerful algorithms.

Binary tree: node 3 only has a left child — .right is None. Max 2 children per node.
pythonclass BinaryNode:
    def __init__(self, value):
        self.value = value
        self.left  = None   # left child
        self.right = None   # right child

Binary Search Tree (BST)

A binary tree with one extra rule: left child < parent < right child. This makes search O(log n).

Real-world analogy

Think of a dictionary. When you search for "mango", you don't start at "a". You open somewhere in the middle and decide: go left (earlier) or right (later)? A BST works identically.

BST with [3,5,7,10,15,20]. Purple dashed = search path for 7. Only 2 comparisons vs scanning all 6 nodes.
pythonclass BST:
    def __init__(self): self.root = None

    def insert(self, value):
        if not self.root: self.root = BinaryNode(value)
        else: self._insert(self.root, value)

    def _insert(self, node, value):
        if value < node.value:
            if node.left is None: node.left = BinaryNode(value)
            else: self._insert(node.left, value)
        else:
            if node.right is None: node.right = BinaryNode(value)
            else: self._insert(node.right, value)

    def search(self, value):
        node = self.root
        while node:
            if value == node.value: return True
            node = node.left if value < node.value else node.right
        return False

bst = BST()
for n in [10,5,15,3,7]: bst.insert(n)
print(bst.search(7))   # True
print(bst.search(9))   # False

What Does O(log n) Actually Mean?

You'll see O(log n) everywhere in computer science. Before going further, let's make it completely concrete — no hand-waving.

The phone book game

Someone picks a number from 1 to 1,000,000. You guess. They say "higher" or "lower". How many guesses do you need? If you always guess the midpoint, you need at most 20 guesses. Not 500,000. Not 1,000,000. Twenty. That's O(log n) in action — each guess eliminates half the remaining possibilities.

O(n) grows linearly with data — O(log n) barely grows at all. That's why BST search is fast and why we care so much about tree height.

The formula is simple: log₂(n) tells you how many times you can cut n in half before reaching 1. With 1 billion nodes, you can always find any value in 30 comparisons — as long as the tree stays balanced.

Why does tree height = number of steps? In a BST, every search starts at the root and goes down exactly one level per comparison. The maximum number of comparisons = the height of the tree. A balanced tree of n nodes has height ≈ log₂(n). An unbalanced one can have height n — turning your O(log n) search into O(n).

Why Balanced Trees? (AVL) — The Full Story

The BST rule (left < parent < right) says nothing about shape. You could have a valid BST that looks like a straight line — and it would be completely useless for fast search. AVL trees enforce shape.

Real-world analogy

Imagine a library organised by Dewey Decimal — that's the BST property (sorted order). Now imagine all the books on the same shelf, arranged in a single long row, versus a multi-shelf bookcase where each shelf holds roughly the same number. Both are "sorted", but finding a book on the balanced shelves takes far fewer steps. AVL enforces the multi-shelf layout automatically.

Sorted insert (1,2,3,4,5) breaks a plain BST into a linked list — O(n) search. AVL auto-rotates to keep the tree balanced — O(log n) guaranteed.

The Balance Factor — AVL's Secret Sensor

Every node in an AVL tree carries one extra number: its balance factor (bf).

concept# balance_factor = height(left_subtree) - height(right_subtree)
#
# bf =  0  → perfectly balanced at this node
# bf =  1  → left side one level deeper — still OK
# bf = -1  → right side one level deeper — still OK
# bf =  2  → left side TWO levels deeper → ROTATE! (left-heavy)
# bf = -2  → right side TWO levels deeper → ROTATE! (right-heavy)
#
# AVL rule: |bf| must be ≤ 1 at EVERY node, always.

What is a Rotation? (The Fix-It Operation)

When inserting a node pushes a balance factor to ±2, AVL performs a rotation — a local restructuring of 3 nodes that restores balance without breaking the BST ordering rule.

A right rotation: the left child (2) takes the root's place. The old root (3) becomes the right child. BST ordering is preserved. Only pointer rewiring — O(1).

There are four rotation cases (left, right, left-right, right-left), but Python's standard library handles all of this for you via the sortedcontainers library. In practice, you use AVL trees without implementing rotations yourself.

Trie — The Autocomplete Tree

A trie stores strings character-by-character. Each edge is a letter. Every root-to-marked-node path spells a complete word.

Trie: every root-to-teal-node path is a complete word. "app", "apple", "application" all share the same a→p→p prefix path.
pythonclass TrieNode:
    def __init__(self):
        self.children = {}      # char → TrieNode
        self.is_end = False   # does a word end here?

class Trie:
    def __init__(self): self.root = TrieNode()

    def insert(self, word):
        node = self.root
        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()
            node = node.children[ch]
        node.is_end = True

    def autocomplete(self, prefix):
        node = self.root
        for ch in prefix:
            if ch not in node.children: return []
            node = node.children[ch]
        return self._collect(node, prefix)

    def _collect(self, node, path):
        results = [path] if node.is_end else []
        for ch, child in node.children.items():
            results.extend(self._collect(child, path+ch))
        return results

trie = Trie()
for w in ["app", "apple", "application", "banana"]: trie.insert(w)
print(trie.autocomplete("app"))  # ['app', 'apple', 'application']

Traversing Trees: DFS & BFS

🔽

DFS — Depth First

Go deep before wide. Uses a stack (or recursion). Three variants: pre/in/post-order.

↔️

BFS — Breadth First

Visit level by level. Uses a queue. Good for shortest path problems.

Same tree, two traversal orders. Circled numbers = visit sequence. DFS dives deep; BFS sweeps level by level.

DFS: Three Orderings

pythondef preorder(node):   # Root → Left → Right
    if not node: return
    print(node.value, end=" ")  # visit BEFORE children
    preorder(node.left); preorder(node.right)

def inorder(node):    # Left → Root → Right
    if not node: return
    inorder(node.left)
    print(node.value, end=" ")  # KEY: on a BST → sorted output!
    inorder(node.right)

def postorder(node):  # Left → Right → Root
    if not node: return
    postorder(node.left); postorder(node.right)
    print(node.value, end=" ")  # visit AFTER children — good for deletion

BFS — Level Order

pythonfrom collections import deque

def bfs(root):
    if not root: return
    queue = deque([root])
    while queue:
        node = queue.popleft()
        print(node.value, end=" ")
        if node.left:  queue.append(node.left)
        if node.right: queue.append(node.right)
When to use which? BFS for shortest path or level-by-level processing. DFS (inorder) for sorted output from BST. DFS (postorder) when children must be processed before parents — like computing directory sizes.

Heaps & Decision Trees

The Heap

A heap is a binary tree where every parent is larger (max-heap) or smaller (min-heap) than its children. Crucially, it's stored as a plain array — no Node objects, no pointers.

Real-world analogy

A hospital ER triage queue. The most critical patient is always at the front, no matter when they arrived. When they leave, the next most critical takes the top spot automatically. That's a min-heap on severity score.

Max-heap (left) vs Min-heap (right / Python heapq). The tree structure is implied by array index math — no pointers needed.
pythonimport heapq

tasks = []
heapq.heappush(tasks, (3, "Low priority task"))
heapq.heappush(tasks, (1, "URGENT: server down"))
heapq.heappush(tasks, (2, "Medium priority bug"))

while tasks:
    priority, task = heapq.heappop(tasks)
    print(f"[{priority}] {task}")
# [1] URGENT: server down
# [2] Medium priority bug
# [3] Low priority task

Decision Tree — The Bridge to ML

A decision tree is a regular tree used for classification. Each internal node asks a yes/no question. Leaf nodes are the final answer.

Decision tree for "should I deploy?". Follow the YES/NO branches based on your data — end at a leaf for the answer.
pythonclass DecisionNode:
    def __init__(self, question=None, yes=None, no=None, answer=None):
        self.question=question; self.yes=yes; self.no=no; self.answer=answer

def decide(node, data):
    if node.answer is not None: return node.answer
    return decide(node.yes if data[node.question] else node.no, data)

tree = DecisionNode(question="tests_pass",
    yes=DecisionNode(question="code_reviewed",
        yes=DecisionNode(answer="✅ Deploy!"),
        no=DecisionNode(answer="⏳ Wait")),
    no=DecisionNode(answer="❌ Fix tests"))

print(decide(tree, {"tests_pass": True, "code_reviewed": True}))
# ✅ Deploy!
When sklearn's DecisionTreeClassifier or XGBoost trains, it builds exactly this structure — but learns the questions automatically from data using Gini impurity or information gain.

Graphs · Foundations

Graphs: Trees Without Rules

A tree is a special case of a graph. A graph is the more general structure — nodes connected by edges, with no restrictions. Cycles are allowed. Multiple paths between nodes are allowed. No root required.

Real-world analogy

A tree is a strict corporate hierarchy — one boss, clear chain. A graph is like LinkedIn — anyone can connect to anyone, mutual connections, clusters, even isolated islands of people.

Three types of graphs. Real graphs can combine all three: a directed, weighted graph (like Google Maps routing).

Three Ways to Represent a Graph in Python

Same 3-node graph represented three ways. Adjacency list wins for most real-world (sparse) graphs.

Dijkstra's Algorithm — Shortest Path

The algorithm behind Google Maps. Given weighted edges, find the shortest path from a source to every other node.

Teal = shortest path found by Dijkstra. Going via C (dist 2) then B (dist 2+1=3) then D (3+5=8) beats the direct A→B→D route (4+5=9).
pythonimport heapq

def dijkstra(graph, start):
    dist = {node: float('inf') for node in graph}
    dist[start] = 0
    heap = [(0, start)]

    while heap:
        d, node = heapq.heappop(heap)
        if d > dist[node]: continue
        for neighbor, weight in graph[node]:
            nd = d + weight
            if nd < dist[neighbor]:
                dist[neighbor] = nd
                heapq.heappush(heap, (nd, neighbor))
    return dist

roads = {
    "A": [("B", 4), ("C", 2)],
    "B": [("C", 1), ("D", 5)],
    "C": [("B", 1), ("D", 8)],
    "D": [],
}
print(dijkstra(roads, "A"))
# {'A': 0, 'B': 3, 'C': 2, 'D': 8}

GNNs · Core Concept

What is a Graph Neural Network?

A regular neural network takes a fixed-size vector as input (like a row in a CSV). But what if your data is a graph — a molecule, a social network? The structure carries meaning that flat vectors can't capture.

Real-world analogy

Imagine describing a person only by their age/income/hobbies (flat vector). Now imagine also knowing who their friends are, and what THEIR friends are like. The graph structure gives context numbers alone can't. GNNs capture this.

The Core Idea: Message Passing

Message passing: neighbors broadcast their features → node 0 averages them → node 0's embedding now reflects its local neighborhood.
python — message passing from scratch (no ML library)import numpy as np

node_features = np.array([
    [1.0, 0.5],  # node 0
    [0.3, 0.8],  # node 1
    [0.7, 0.2],  # node 2
    [0.4, 0.6],  # node 3
])
neighbors = {0: [1,2,3], 1: [0], 2: [0], 3: [0]}

def message_passing_round(features, neighbors):
    new_features = np.zeros_like(features)
    for node, nbrs in neighbors.items():
        agg = features[nbrs].mean(axis=0)        # AGGREGATE
        new_features[node] = 0.5*features[node] + 0.5*agg  # UPDATE
    return new_features

h1 = message_passing_round(node_features, neighbors)
print(h1[0])  # [0.62, 0.52] — blended with neighbor features
After K rounds of message passing, each node's embedding reflects the entire K-hop subgraph around it. This is what GNNs learn to do — but with learned weight matrices instead of fixed averaging.

GNNs · Architectures

GCN, GAT & GraphSAGE

GCN — Graph Convolutional Network

The foundational GNN (2017). Each node updates its embedding by taking a normalized weighted average of its neighbors, then applying a learned linear transformation.

GCN: two layers = each node sees info from 2-hop neighborhood. The W matrices are the parameters learned during training.

GAT — Graph Attention Network

GCN treats all neighbors equally. GAT adds attention weights — it learns which neighbors matter more.

GAT: each neighbor gets an attention score α. Node j1 (α=0.7) influences node i much more than j3 (α=0.1). Weights are learned, not fixed.

GNN Tasks & Training

Three Levels of GNN Predictions

Three GNN task types. Node: classify each node independently. Edge: predict connections. Graph: classify the entire graph (e.g. molecular toxicity).
python — training loopimport torch.nn.functional as F

def train(model, data, optimizer):
    model.train(); optimizer.zero_grad()
    out = model(data.x, data.edge_index)
    # Only compute loss on LABELED nodes — GNN still learns from
    # the unlabeled nodes' graph structure via message passing!
    loss = F.cross_entropy(out[data.train_mask], data.y[data.train_mask])
    loss.backward(); optimizer.step()
    return loss.item()

def evaluate(model, data):
    model.eval()
    with torch.no_grad():
        pred = model(data.x, data.edge_index).argmax(dim=1)
        acc = (pred[data.test_mask] == data.y[data.test_mask]).float().mean()
    return acc.item()
Over-smoothing: If you stack too many GNN layers, every node's embedding converges to the same value — the tree grew too deep and all nodes look the same. In practice, 2–4 layers is optimal.

PyTorch Geometric (PyG) is the standard library for GNNs. It handles graph data structure, efficient message passing, and comes with 60+ benchmark datasets.

The Data Object

PyG Data object. edge_index uses COO (coordinate) format — two rows: sources and targets. For undirected graphs, add both directions.

Complete GCN on Real Dataset

python — full working GNN (Cora citation network)import torch, torch.nn.functional as F
from torch_geometric.nn import GCNConv
from torch_geometric.datasets import Planetoid

# Download Cora: 2708 papers, 5429 citation edges, 7 categories
dataset = Planetoid(root='/tmp/Cora', name='Cora')
data = dataset[0]

class GCN(torch.nn.Module):
    def __init__(self, in_channels, hidden, out_channels):
        super().__init__()
        self.conv1 = GCNConv(in_channels, hidden)
        self.conv2 = GCNConv(hidden, out_channels)

    def forward(self, x, edge_index):
        x = F.relu(self.conv1(x, edge_index))   # 1-hop neighborhood
        x = F.dropout(x, p=0.5, training=self.training)
        x = self.conv2(x, edge_index)             # 2-hop neighborhood
        return F.log_softmax(x, dim=1)

model = GCN(dataset.num_features, 16, dataset.num_classes)
optimizer = torch.optim.Adam(model.parameters(), lr=0.01, weight_decay=5e-4)

for epoch in range(200):
    model.train(); optimizer.zero_grad()
    out = model(data.x, data.edge_index)
    loss = F.nll_loss(out[data.train_mask], data.y[data.train_mask])
    loss.backward(); optimizer.step()

model.eval()
pred = model(data.x, data.edge_index).argmax(dim=1)
acc = (pred[data.test_mask] == data.y[data.test_mask]).float().mean()
print(f"Accuracy: {acc:.4f}")  # ~81%

Project 1 — Fraud Detection on a Transaction Graph

Problem: You have a CSV of bank transactions. Each row is a payment from one account to another. Flag fraudulent accounts. This is a node classification problem on a directed, weighted graph.

Why a graph beats a flat model

A fraudster rarely acts alone. They create rings of accounts that pass money between each other. A flat ML model sees each account independently. A GNN sees that account X transacts exclusively with 3 other accounts that all transact with each other — that ring structure is the signal.

    def __init__(self):

        x = F.dropout(x, p=0.3, training=self.training)

# Train / val / test split (stratified by label)

Project 3 — Python Package Dependency Tree (Trees + Graphs Together)

Problem: Given all Python packages on PyPI, build a dependency graph. Find circular dependencies, rank packages by centrality, find the shortest upgrade path. This is a pure graph algorithm problem — no ML needed. A great bridge project between Chapters 5 and 6.

python — build PyPI dependency graph with networkximport subprocess, json
import networkx as nx
import matplotlib.pyplot as plt

def get_deps(package: str) -> list:
    """Get direct dependencies of a pip package"""
    result = subprocess.run(
        ["pip", "show", package], capture_output=True, text=True
    )
    for line in result.stdout.splitlines():
        if line.startswith("Requires:"):
            deps = line.replace("Requires: ", "").split(", ")
            return [d.strip() for d in deps if d.strip()]
    return []

def build_dep_graph(root_package: str, depth: int = 2) -> nx.DiGraph:
    """BFS from root package — this is literally graph BFS from Chapter 5"""
    G = nx.DiGraph()
    queue = [(root_package, 0)]
    visited = set()
    while queue:
        pkg, d = queue.pop(0)
        if pkg in visited or d > depth: continue
        visited.add(pkg)
        deps = get_deps(pkg)
        for dep in deps:
            G.add_edge(pkg, dep)    # directed: pkg depends on dep
            queue.append((dep, d+1))
    return G

G = build_dep_graph("torch", depth=2)

# Find cycles (circular dependencies — bad!)
cycles = list(nx.simple_cycles(G))
print(f"Circular dependencies found: {len(cycles)}")

# PageRank: which packages are most "important"?
pr = nx.pagerank(G)
top5 = sorted(pr.items(), key=lambda x: -x[1])[:5]
print("Most depended-upon packages:", top5)

# Shortest upgrade path from old_pkg to new_pkg
try:
    path = nx.shortest_path(G, "numpy", "filelock")
    print(f"Dependency chain: {' → '.join(path)}")
except nx.NetworkXNoPath:
    print("No dependency path")

Project 4 — Molecule Property Prediction (Graph Classification)

Problem: Given molecules represented as graphs (atoms = nodes, bonds = edges), predict whether a molecule is toxic. This is the canonical graph classification GNN benchmark and the most common GNN application in drug discovery.

python — full molecule GNN with OGB benchmark datasetimport torch
import torch.nn.functional as F
from torch_geometric.nn import GINConv, global_mean_pool
from torch_geometric.loader import DataLoader
from ogb.graphproppred import PygGraphPropPredDataset, Evaluator

# OGB: Open Graph Benchmark — standardized, reproducible
# ogbg-molhiv: 41,127 molecules, task: HIV inhibition yes/no
dataset = PygGraphPropPredDataset(name="ogbg-molhiv")
split_idx = dataset.get_idx_split()

train_loader = DataLoader(dataset[split_idx["train"]], batch_size=64, shuffle=True)
val_loader   = DataLoader(dataset[split_idx["valid"]], batch_size=64)
test_loader  = DataLoader(dataset[split_idx["test"]],  batch_size=64)

# GIN is the standard choice for graph-level molecular tasks
# (more expressive than GCN — proven by the Weisfeiler-Lehman test)
class MolGIN(torch.nn.Module):
    def __init__(self, hidden=64):
        super().__init__()
        nn1 = torch.nn.Sequential(torch.nn.Linear(9, hidden), torch.nn.ReLU(), torch.nn.Linear(hidden, hidden))
        nn2 = torch.nn.Sequential(torch.nn.Linear(hidden, hidden), torch.nn.ReLU(), torch.nn.Linear(hidden, hidden))
        self.conv1 = GINConv(nn1)
        self.conv2 = GINConv(nn2)
        self.classifier = torch.nn.Linear(hidden, 1)

    def forward(self, x, edge_index, batch):
        x = F.relu(self.conv1(x.float(), edge_index))
        x = F.relu(self.conv2(x, edge_index))
        # global_mean_pool: collapse all nodes → one graph embedding
        # batch tells PyG which nodes belong to which graph in the batch
        x = global_mean_pool(x, batch)
        return self.classifier(x)

model = MolGIN()
optimizer = torch.optim.Adam(model.parameters(), lr=0.001)
evaluator = Evaluator(name="ogbg-molhiv")  # uses ROC-AUC, not accuracy

for epoch in range(50):
    model.train()
    for batch in train_loader:
        optimizer.zero_grad()
        pred = model(batch.x, batch.edge_index, batch.batch)
        loss = F.binary_cross_entropy_with_logits(pred, batch.y.float())
        loss.backward(); optimizer.step()

# OGB evaluator gives ROC-AUC — the right metric for drug discovery
# (you care about ranking, not just binary classification threshold)
print("Evaluation uses ROC-AUC, not accuracy")
print("State-of-art baseline: ~0.77 ROC-AUC. Simple GIN: ~0.70")
